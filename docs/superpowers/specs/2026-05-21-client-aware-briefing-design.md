# Daily News v2 — Client-Aware Briefing

**Date:** 2026-05-21
**Author:** Midya (designed in conversation with Kai)
**Status:** Approved for implementation planning
**Scope:** Python pipeline only (`src/daily_news/`, `config/`, `scripts/`, `tests/`). No changes to the Expo app under `app/`.

---

## Problem

The Daily News briefing today scores and summarizes news against a static topic taxonomy (8 sections, fixed keywords). It does not know that Midya is now running an active consulting practice with named clients, a Notion-managed prospect pipeline, defined methodologies, and a 2026-2027 goal to establish herself as an AI thought leader. As a result:

1. Stories materially affecting her active engagements (Capilano University, Mogul Realty Group, Cushman & Wakefield Edmonton) are missed because no search ever fetches them and no prompt ever favors them.
2. AI-section picks are generic AI news, not AI news filtered through her professional lens (marketing / higher-ed enrollment / real estate ops / public-institution governance) or her thought-leadership lens.
3. The "why this matters" commentary is generically executive-flavored, not engagement-aware.

## Goal

Make the morning briefing meaningfully sharper for Midya's current business reality, in a way that automatically adapts as her client roster and prospect pipeline change — without leaking client relationships in the published output.

## Non-goals

- Changing the audio voice, TTS provider, or delivery channel.
- Changing the dedup/scoring/recency formula math.
- Touching the Expo mobile app.
- Building a private/auth'd briefing channel (explicitly deferred — keep on GitHub Pages, neutral framing handles privacy).
- Per-day length increase. Existing 1600–2400 word audio target stands.

---

## Approach (hybrid: static prompt + dynamic watchlist)

Stable facts about Midya's business identity bake into the system prompt. Volatile facts (current client roster, prospect pipeline) load from `config/watchlist.yaml`, which is regenerated by a local sync script that reads `~/Desktop/Clients/*/CLAUDE.md` + Notion 🎯 Leads.

```
                  STATIC (system prompt)        DYNAMIC (user message)
                  ─────────────────────         ──────────────────────
                  Brand identity                config/watchlist.yaml
                  Industry lanes                  ├─ clients
                  Methodologies                   ├─ prospects (Notion)
                  Thought-leadership lens         ├─ peer_orgs
                  Section structure (9)           └─ thought_leadership_themes
                  Framing rules                       (curated locally)
                          │                                 │
                          └─────────────────┬───────────────┘
                                            ↓
                            Claude call (summarize.py)
                                            ↓
                            Briefing JSON: 9 sections
                            (Watchlist is new #1)
                                            ↓
                            HTML + audio (existing render path)
```

### Key design choices

| Choice | Rationale |
|--------|-----------|
| Hybrid (not fully dynamic) | Static facts rarely change; volatile ones change weekly. Mirrors the structure where it serves caching + readability. |
| Watchlist is silent metadata | The new section discusses watched orgs as "industry coverage," never as Midya's clients/prospects. Solves the GitHub Pages public-URL privacy risk without forcing a host migration. |
| Org-name search expansion (orgs only, not people) | Without injected searches, the candidate pool never contains client-named stories and the new section stays empty. People-name searches were ruled out as too noisy. |
| Sync runs locally, not in CI | GitHub Actions runner has no access to `~/Desktop/Clients/`. Combining the Notion fetch into the same script keeps the workflow simple. |

## Components

### 1. `config/watchlist.yaml` (new, generated)

Committed to the **private** daily-news repo. Schema:

```yaml
clients:
  - org: "Capilano University"
    aliases: ["CapU", "Capilano U"]
    industry: higher_ed_canada
    posture: active            # active | wrapping | past
    notes: "Public BC post-secondary; SEM pilot"

prospects:
  - org: "<Lead Org from Notion 🎯 Leads>"
    industry: higher_ed_canada
    stage: discovery

peer_orgs:
  - org: "AACRAO"
    relationship: consortium_peer
  - org: "Generation Marketing"
    relationship: consortium_peer
  - org: "Illume Student Advisory Services"
    relationship: consortium_peer

thought_leadership_themes:
  - "AI in marketing operations"
  - "AI in higher-ed enrollment / SEM"
  - "AI in real estate (agent productivity, brokerage ops)"
  - "Public-institution AI governance (FIPPA, BC data residency)"
  - "AI Search Readiness / generative search visibility"
```

Aliases enable disambiguation (e.g., "Mogul Realty Group" OR "Mogul RG" as a single quoted-OR Google News query).

### 2. `scripts/sync_watchlist.py` (new)

Local-only script. Run when the roster changes (weekly cadence sufficient).

Inputs:
- Glob: `~/Desktop/Clients/*/CLAUDE.md`
- Notion API (token at `~/.claude/.notion-token`), 🎯 Leads database
- Hand-curated `config/peer_orgs.yaml`
- Hand-curated `config/thought_leadership_themes.yaml`

Behavior:
1. For each client folder: extract org name from the H1, classify industry by keyword match against the CLAUDE.md body, determine posture from the Status block.
2. Notion: fetch all 🎯 Leads not in stage `lost` or `closed-won`. Extract org name + stage + classified industry.
3. Merge with peer_orgs.yaml + thought_leadership_themes.yaml.
4. Write `config/watchlist.yaml` with the timestamp + source attribution header.

Failure modes:
- Notion API down → keep last-good prospects from existing watchlist.yaml; log warning. Don't blow away with empty list.
- Client folder missing CLAUDE.md → skip with warning.
- Sync run with no clients → emit a watchlist with empty clients/prospects sections but valid YAML. Pipeline must tolerate this.

### 3. `src/daily_news/summarize.py` (modified)

**Add `"watchlist"` to `_TOPIC_KEYS`** and to the `topic_key` enum in `_DIGEST_SCHEMA`.

**Replace the `WHO MIDYA IS` block** with the expanded brand identity (Midya U Advisory, two industry lanes, methodologies, husband's brokerage context, AI thought-leadership goal).

**Replace the AI-section directive** with the dual-lens version (professional lens + thought-leadership lens; explicit skip rules).

**Insert a new section #1 spec** for "Watchlist" — with the **framing rule** as a hard constraint ("never describe these orgs as your client/prospect/peer; write as a sharp news editor who happens to track them for industry coverage reasons").

**Renumber existing sections 1–8 to 2–9.** The 8-section briefing structure ordering is preserved relative to itself; only Watchlist is inserted ahead.

**Inject `WATCHLIST_ORGS` block into the user message**, built at runtime from `watchlist.yaml`. Placed before the `candidates` JSON.

### 4. `src/daily_news/fetch.py` (modified)

`load_searches()` (or equivalent path that builds the search list) becomes:

```python
searches = load_yaml("config/searches.yaml") + build_watchlist_searches(watchlist, cap=25)
```

`build_watchlist_searches(watchlist, cap=25)`:
- For each org in `clients + prospects + peer_orgs`, build a single Google News query: `'"<primary>" OR "<alias1>" OR ...'` (quoted exact phrases).
- Tag each generated search with topics `[watchlist, <industry>]`.
- Credibility: 0.70 (Google News default, matches existing convention).
- Prioritize clients > prospects > peers; cap total dynamic searches at 25 (warn + drop overflow with log line naming dropped orgs).

### 5. `config/interests.yaml` (modified)

Add new topic:

```yaml
watchlist:
  weight: 1.5    # boost above normal topics so watchlist hits clear the candidate pool
  keywords: []   # source-tagging from fetch.py does the work
```

### 6. `templates/digest.html` + `static/style.css` (modified)

- New colour token for `watchlist` topic. Suggested: muted slate-blue or deep-green. Final colour decision deferred to implementation.
- Watchlist section auto-renders via existing section iteration (no template-structure change needed once topic_key is added).
- Section nav at top includes Watchlist anchor.

### 7. Audio chapter handling

No code change. Existing `_MARKER_RE` regex already matches `[[SECTION:<topic_key>]]`. Adding `watchlist` to `_TOPIC_KEYS` enum is sufficient. Prompt instructs Claude to emit `[[SECTION:watchlist]]` like any other section.

## Data flow per run

```
1. GitHub Actions cron fires (13:30 UTC)
2. pipeline.py loads:
     - config/feeds.yaml
     - config/searches.yaml
     - config/interests.yaml
     - config/watchlist.yaml             ← NEW
3. fetch.py:
     - feeds + static searches + dynamic watchlist searches (capped at 25)
4. dedup → score (watchlist topic weighted 1.5)
5. summarize.py:
     - System prompt: static (expanded WHO MIDYA IS, dual-lens AI, watchlist spec)
     - User message: candidates + WATCHLIST_ORGS block
6. render → HTML, MP3, RSS → publish to gh-pages
```

`watchlist.yaml` updates only happen via local sync — CI never modifies it.

## Privacy & framing (load-bearing)

The published audio + HTML are on a public-URL GitHub Pages site (unlisted, not authenticated). The Watchlist section discusses watched orgs as news subjects on their public-news merits. The prompt forbids:

- "Your client" / "your prospect" / "your peer" framing
- Any phrase implying Midya has a relationship to the org
- Any reference to private engagement context (deal stage, internal stakeholders, methodology applied)

Watched orgs can be named (including Mogul Realty Group — confirmed acceptable by Midya). The prohibition is on **relationship framing**, not on naming. Read like a sharp industry editor, not a CRM-fed bot.

## Migration / rollout

1. Land all code changes behind a single PR.
2. Run `scripts/sync_watchlist.py` locally; commit the initial `config/watchlist.yaml`.
3. Run `python -m daily_news run --mode dry` locally to verify candidate pool now contains watchlist-tagged articles.
4. Run full pipeline locally once; eyeball the rendered HTML + audio to confirm framing rule is respected.
5. Push to main; next cron run picks it up.

Rollback: revert PR. `watchlist.yaml` becoming missing is a soft-fail (warning + skip), so partial rollback (yaml only) is safe.
